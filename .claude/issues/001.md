## 概要
Claude Code (Opus 4.7/4.8) で、モデルが壊れた `tool_use` ブロックを生成し、ハーネスが
"The model's tool call could not be parsed" としてターンごと破棄する不具合がある。結果、
ツールが発火せずアシスタントが無言で停止し、可視テキストに迷子トークン(`court` / `course`
/ `call` 等)とツール引数の断片が漏れる(通称「オモラシ」)。

本 issue は、この破損を Stop hook で検知し「続けてください」と最小フィードバックで自動継続
させるリカバリ機構を実装し、単一スクリプト＋インストーラーの OSS リポジトリ
`cchook-recovery`(MIT) として公開可能な状態に整える。

## 詳細
### 背景・経緯
- 当初 `/loop` のメールチェックで迷子トークンを観測。原因調査の結果、`/loop` 非依存で通常会話中にも発生。
- 仮説検証: 1M コンテキスト無効化(`CLAUDE_CODE_DISABLE_1M_CONTEXT=1`)・`effortLevel: low` の双方を試すも
  解消せず。→ コンテキスト量や thinking ではなく **Opus 系の tool_use シリアライズ崩れ**が主因と判断。
- deep-research の結論: 「Stop hook による auto-continue」自体は既存(ralph-wiggum 等)だが、
  **tool_use 崩れ/漏れトークンを“構造的に”検知して回復する特化型は空白領域**であり OSS 公開の価値あり。

### 検知と二重ガードのフロー
```mermaid
flowchart TD
    A[Stop hook 発火] --> B{last_assistant_message に<br/>漏れトークン単独行?}
    B -- あり --> D[検知]
    B -- なし --> C{transcript:<br/>stop_reason=tool_use かつ<br/>tool_use ブロック無し?}
    C -- はい --> D
    C -- いいえ --> Z[クリーン停止<br/>状態リセットして終了]
    D --> E{断片が前回と変化?}
    E -- 変化 --> F[連続カウンタ count=0 に]
    E -- 同一 --> G[count 据え置き]
    F --> H{count>=MAX(3)<br/>または total>=HARD_MAX(8)?}
    G --> H
    H -- 到達 --> I[諦め: systemMessage で手動継続を促す]
    H -- 未到達 --> J[count++/total++ 記録<br/>BACKOFF 後 decision:block<br/>reason='Please continue.']
```

### 検知シグネチャ
1. **漏れトークン検知**: `LEAK_WORDS_RE='court|course|cource|call'` の単独行マッチ(高速・経験則上確実・拡張容易)
2. **構造検知**: 直近 assistant が `stop_reason=tool_use` かつ tool_use ブロック数 0 (#61133)。
   漏れ単語の種類に依存せず捕捉できる一般化シグネチャ。transcript 不読時は (1) にフォールバック。


## 方針
- Stop hook (`~/.claude/hooks/toolparse_recovery.sh`) として実装。
- 検知時に `{"decision":"block","reason":"Please continue."}` を返し同一ターン内で継続させる。
- 無限ループ防止に **二重ガード**: 同一断片の連続上限 `MAX=3` / セッション累積上限 `HARD_MAX=8`。
- 断片が変われば「別インシデント」として連続カウンタをリセット。クリーン停止で全リセット。
- 一時的崩れ対策に `BACKOFF_SEC=2` の軽いバックオフ。キルスイッチ `CLAUDE_NO_TOOLPARSE_RECOVERY=1`。
- 発動を JSONL でログ(`logs/toolparse_recovery.log`、`count`/`total`/`fragment`/`cwd`)。
